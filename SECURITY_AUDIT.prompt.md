# SECURITY AUDIT PROMPT

Paste-ready application security review prompt for AI rules or custom instructions.

```markdown
Perform a detailed security review of the entire web application and all publicly exposed APIs. Treat this as a real application security assessment performed by a senior application security analyst and pentester reviewing production-grade code, request flows, authorization logic, data models, middleware, input validation, SSR behavior, caching, file handling, integrations, and public edge components. The stack to assess includes a Vue 3 frontend, Nuxt 4 with SSR for SEO, a Python FastAPI backend, JWT-based authentication, human-friendly URLs where a readable slug ends with a suffix such as `-xxxx` that is parsed by the backend and used to resolve a record, and any public traffic that passes through Cloudflare Workers or equivalent edge logic. Ignore and do not report missing CAPTCHA because it will be added at the end of the project, ignore and do not report missing rate limiting because it is handled by Cloudflare, and ignore and do not report hardcoded secrets in application code because secrets are injected through Kubernetes. Do not dilute the review with generic advice, do not write a shallow checklist, do not invent findings without evidence, and do not rely on frontend assumptions when backend enforcement is required.

Write the final report in English, in a technical but readable tone, with concrete findings grounded in code, request handling, or clearly inferable behavior. The report should read like a professional application security assessment, not like a brainstorming dump or a raw taxonomy checklist. Prefer short, well-separated sections, compact metadata lines, and structured finding blocks that are easy to scan quickly. Do not produce giant uninterrupted walls of text and do not turn findings into essay-style prose. Present confirmed bugs as a clear list of findings sorted by severity, and make every finding self-contained. Each finding must show the exact problematic code location whenever it is available from the reviewed material, including file paths, symbols, and line references where possible. If a class of vulnerability is not present, explicitly say why it appears mitigated, based on specific code patterns, middleware, validation, or runtime behavior. If something cannot be fully confirmed from the available code or configuration, place it in an explicitly marked unverified area rather than turning uncertainty into a finding.

# Broken Access Control

Review authorization comprehensively across all routes, service methods, background flows, and edge-mediated traffic, and verify that access decisions are enforced on the backend for every sensitive action rather than merely implied by frontend navigation or hidden UI elements. Inspect whether a logged-in but unauthorized user can read, create, update, delete, publish, unpublish, archive, restore, or otherwise manipulate resources belonging to another user or another organization, and verify that every operation involving listings, organizations, memberships, invitations, attachments, drafts, settings, analytics, moderation states, and internal records performs server-side ownership or role checks before returning data or mutating state. A realistic bug would appear when an endpoint accepts a valid JWT and a resource identifier, then returns HTTP 200 with another tenant’s object because the code only checks that the caller is authenticated, not that the caller is allowed to operate on that specific object. Another common manifestation is inconsistent behavior where one endpoint correctly filters by `organization_id` but a neighboring detail, preview, export, or update endpoint does not. Also examine whether authorization depends on fragile client-provided context such as a selected organization value, a hidden field, or a header that should never be trusted from the browser.

# IDOR / BOLA

Inspect all direct and indirect object references, including numeric IDs, UUIDs, slug fragments, suffixes extracted from SEO-friendly URLs, organization IDs, invitation IDs, file IDs, asset keys, nested object identifiers in JSON, and any internal references passed through query parameters or headers. Because this application uses readable URLs with a final suffix used by the backend to resolve records, pay special attention to whether changing only the suffix, the slug, or the separator allows access to another record, or whether the backend trusts the extracted identifier without rechecking tenant boundaries and ownership. A real issue would usually present as successful retrieval or modification of another user’s resource with HTTP 200 or 204 after replacing the object reference, but it can also appear as more subtle object leakage through list endpoints, relationship endpoints, or preview endpoints that return partial metadata. Check both read and write operations, because some systems block direct reads while still allowing unauthorized edits, uploads, deletions, or state changes by referencing a foreign object. Confirm whether object-level authorization is consistently enforced in every microservice and any worker-mediated route, not just on the primary CRUD endpoints.

# Authentication Bypass

Assess whether any path exists that allows access to protected functionality without valid authentication or with weakened authentication, including alternate route prefixes, forgotten legacy endpoints, edge routes, worker passthroughs, debug interfaces, preflight-handling mistakes, undocumented API versions, SSR data loaders, or internal routes accidentally exposed to the internet. In FastAPI-based systems, this often happens when dependencies that enforce user context are attached to some routers but omitted from others, or when helper endpoints are created for internal use and never wrapped with the same security middleware as the main API. A typical exploit would be a request that should return 401 or 403 but instead returns 200, perhaps because the route is mounted twice, because the worker forwards it around a security check, or because a fallback code path is taken when authentication fails. Also verify whether the system trusts caller-controlled headers such as user identifiers, organization identifiers, or role hints in place of deriving identity exclusively from a validated token and backend-side membership lookup.

# Privilege Escalation

Review all role-sensitive paths to determine whether a normal user can obtain more privileges than intended, whether directly by changing fields such as `role`, `is_admin`, `owner_id`, `organization_id`, `permissions`, or `status`, or indirectly through invitation workflows, ownership transfer logic, group membership edits, account linking, or administrative edge cases. The most dangerous bugs here are often not full authentication failures but subtle logic errors where the code assumes that anyone who can edit a record may also edit privileged fields on that record, or where self-service profile updates accidentally expose administrative columns. A realistic sign of the issue would be that a user can send a crafted JSON payload to an update endpoint and receive a success response while their privileges, tenant scope, publication rights, or moderation rights silently expand. Also inspect scenarios involving self-demotion, removal of the last administrator, reassignment of organization ownership, and indirect privilege gains through imported data, duplicated objects, invitation acceptance, or stale JWT claims that no longer match database reality.

# Mass Assignment

Examine create and update endpoints for overly broad request models, especially where incoming JSON is mapped directly to ORM models, domain entities, or update dictionaries without a strict allowlist of writable fields. In FastAPI this risk often appears when permissive Pydantic models accept fields that the frontend never sends but the backend still honors, allowing a caller to inject hidden properties such as `owner_id`, `organization_id`, `visibility`, `published`, `verified`, `moderation_state`, `internal_note`, `billing_plan`, `created_by`, `deleted_at`, or other server-controlled values. The bug is real when the API returns success after receiving extra fields that should be ignored or rejected, and the stored object ends up reflecting attacker-supplied internal properties. Even if the UI has no form fields for those values, the vulnerability still exists if a crafted request can persist them. Review nested objects, arrays of child records, partial update semantics, merge logic, and serializer behavior, because mass assignment frequently hides inside bulk updates and relationship-editing endpoints rather than obvious profile edit forms.

# Insecure Direct Object Reference

Treat this section as a focused inspection of raw object references wherever the code binds a request parameter directly to data access, even if broader object-level authorization has already been reviewed. Check whether identifiers appearing in the path, query string, request body, relation fields, file names, signed URLs, image references, or generated links are used to retrieve or mutate records without verifying that the caller has the right to access that exact object in that exact context. In this application, special attention should be paid to endpoints that parse the last token from a slug-like URL and then query the database, because a developer might assume the slug is just cosmetic while the suffix becomes the real authority for record resolution. A practical bug would surface if replacing the suffix or the raw ID causes the backend to return another tenant’s resource, even if the human-readable slug itself no longer matches. Confirm whether the system cross-checks both the resolved object and the caller’s current tenant, role, and relationship to that object before performing the action.

# Security Misconfiguration

Inspect the application, runtime, framework, and edge configuration for insecure defaults, exposed debug surfaces, inconsistent trust boundaries, and deployment-time mistakes that create attack opportunities even when business logic appears correct. Review whether production still exposes interactive API documentation, stack traces, excessive OpenAPI schemas, internal-only routes, debug logging, development CORS settings, permissive trusted host behavior, unsafe reverse proxy header handling, or SSR behavior that embeds data not meant for unauthenticated or cross-tenant use. In a Nuxt SSR and FastAPI environment, a real issue might be that server-rendered pages include serialized application state containing private user information, or that the backend exposes Swagger UI and all operation schemas on a public host. Misconfiguration also includes unsafe cache directives, permissive content-type handling, unvalidated forwarded headers, accidental public storage buckets, weak cookie attributes if any cookies exist, and mismatches between Cloudflare Worker behavior and origin expectations. Distinguish between harmless defaults and materially risky exposure by tying each observation to a realistic exploit path or data leak.

# Sensitive Data Exposure

Analyze every response surface, including JSON API responses, SSR HTML, embedded hydration state, meta tags, error pages, redirects, logs shown to the client, and object serializers, to determine whether the application reveals more data than a caller should see. This includes personal data, tenant identifiers, private email addresses, phone numbers, hidden status flags, internal moderation states, billing information, signed file URLs, internal notes, soft-deleted records, relationship metadata, or diagnostic details that help an attacker pivot. A typical bug is an API that returns an entire ORM object or internal DTO when the UI only needs a public subset, leading to sensitive fields being sent over the wire even if the frontend never renders them. Another variant is SSR output that serializes private data into page state for one user and then makes it accessible to caches, crawlers, or later visitors. Look not only for obviously secret values but for contextual information that enables user enumeration, organization discovery, or relationship mapping, because small leaks often compound into much more serious access-control or phishing attacks.

# JWT Vulnerabilities

Review the entire JWT trust model, including signature verification, accepted algorithms, issuer and audience validation, expiration handling, subject mapping, refresh flows, and the use of claims for tenant or role decisions. Confirm that the backend does not accept unsigned or incorrectly signed tokens, does not allow algorithm confusion, and does not trust claims such as organization membership, role, or feature entitlement longer than the underlying database relationship remains valid. A realistic weakness would be a backend that accepts a token with a valid structure but the wrong issuer, or one that trusts `organization_id` or `role` from the token payload after the user has been removed from that organization or demoted. Another common bug is overlong token lifetime, missing refresh token rotation, lack of revocation after logout or password reset, or reuse of refresh tokens without replay detection. Examine how tokens are stored or forwarded through SSR and API boundaries, whether they appear in logs or redirects, and whether any worker or middleware layer transforms authentication headers in a way that can be spoofed or stripped.

# Session Management Issues

Even though the application uses JWT, assess session management as a whole rather than treating token validation alone as sufficient. Determine how login state is created, renewed, revoked, and invalidated when users log out, change passwords, rotate email addresses, lose organization membership, or have their accounts suspended or deleted. A genuine issue exists if a previously valid token continues to authorize actions after the user’s permissions should have changed, or if concurrent sessions remain active indefinitely without any server-side invalidation model. Investigate whether access tokens and refresh tokens are stored in a way that increases exposure, whether stale tokens survive privilege changes, whether refresh flows can be replayed, whether old sessions remain valid after sensitive account events, and whether SSR components accidentally persist one user’s authentication context into another request. If the backend derives authorization entirely from token claims without checking current database truth for critical operations, document that as a session-state design weakness even when the token itself is cryptographically valid.

# CSRF

If any part of the application uses cookies, browser-managed session state, implicit credentials, or hybrid auth patterns around SSR, admin tooling, or edge layers, assess cross-site request forgery thoroughly even if the main API typically expects bearer tokens. Review every state-changing operation to determine whether the browser could automatically send credentials to it from a malicious origin, and whether the application enforces CSRF tokens, SameSite protections, and Origin or Referer validation where appropriate. A real bug would be a request that performs a meaningful action while authenticated via browser context, despite originating from an attacker-controlled page, and the observable result might be a silent HTTP 200 or 302 on a change endpoint that should have required an anti-CSRF control. Also inspect flows such as invitation acceptance, logout, profile changes, email changes, password resets, content publishing, and any SSR-powered form submissions, because those are common places where developers forget that browser credential behavior differs from raw API clients.

# SQL Injection

Inspect every place where user input influences database queries, whether through raw SQL, ORM helpers, dynamic filtering, search syntax, sorting, pagination, date ranges, reporting endpoints, exports, admin tools, or utility queries buried outside the main controller flow. Focus especially on dynamic `ORDER BY`, `WHERE`, `LIMIT`, and search clauses, because developers often parameterize values but not identifiers or SQL fragments. A real vulnerability may manifest as SQL syntax errors in responses, unexpected data access, timing differences on crafted payloads, or the ability to alter filtering or sorting behavior beyond the intended set of fields. In FastAPI systems, SQL injection can hide in repository helpers that concatenate strings based on query parameters, as well as in raw analytics queries that bypass normal ORM safety. Verify whether every user-controlled field is validated against strict allowlists before influencing query structure, and distinguish safe parameterization from partial safety where one untrusted fragment still controls table names, column names, operators, or whole clauses.

# NoSQL Injection

If any part of the system uses document stores, search engines, JSON-like query objects, dynamic filter builders, or externally backed search services, verify that user-controlled input cannot become part of query operators, logical expressions, or regular expressions beyond the intended data type. This issue often appears when APIs accept flexible JSON filters and pass them to a backend search or storage layer with insufficient schema enforcement, allowing an attacker to provide operator objects instead of primitive values. A practical example would be a field expected to be a string but accepted as an object containing comparison or negation semantics, resulting in a wider match set or authentication bypass in a login or search flow. Even if the primary database is SQL, this review should cover auxiliary storage or search mechanisms, including analytics stores, full-text search endpoints, and document-like caches. Confirm whether Pydantic models and downstream conversion logic truly enforce primitive types and disallow unexpected nested structures, rather than merely trusting what the frontend happens to send.

# Command Injection

Review whether any endpoint, background task, file-processing path, export feature, image pipeline, OCR job, PDF generator, archive builder, deployment helper, or third-party CLI wrapper executes shell commands or system processes using data that could be influenced by the user. A severe bug would exist if a filename, URL, filter expression, path, template name, or command option is interpolated into a shell string or passed unsafely to a subprocess, enabling metacharacter injection or unintended command execution. In practice, the observable signal may be unusual process output, command errors leaking into logs or HTTP responses, or the ability to alter the behavior of tools invoked by the backend. Because image conversion, compression, map-related tooling, PDF generation, and document handling often rely on external binaries, inspect those paths carefully. Distinguish safe use of argument arrays and strict parameter validation from dangerous shell invocation, and trace whether user input can reach any process execution boundary directly or through helper functions that hide the risk.

# Server-Side Template Injection (SSTI)

Examine any server-side rendering or templating beyond standard Vue/Nuxt output, including email templates, PDF or document generation, report rendering, dynamic landing page content, notification messages, CMS-like blocks, or helper features that interpret placeholders. SSTI becomes a real vulnerability when user-controlled input is treated as part of a template language rather than as inert data, allowing the template engine to evaluate expressions, access internal objects, or sometimes execute code. A practical sign would be that strings resembling template syntax are rendered dynamically rather than escaped, that certain payloads change output structure, or that error messages reveal template engine internals when malformed input is submitted. Even if the main frontend is safe, backend-generated HTML or email content can still be vulnerable if untrusted content is injected into the template source itself rather than passed as data. Inspect how templates are constructed, whether user input can influence includes or expressions, and whether rendering helpers ever call unsafe evaluation primitives.

# XML External Entity (XXE)

Review all import, upload, parsing, and transformation paths that may handle XML or XML-derived formats, including SVG, office documents, feed imports, configuration-like uploads, metadata extraction, and any third-party parser invoked by the application. XXE exists when external entities, DTD processing, or external resource resolution are allowed in a parser that receives attacker-controlled content, potentially leading to local file disclosure, server-side request forgery, parser denial of service, or unexpected network access. A realistic signal would be that malformed XML causes parser errors revealing entity processing behavior, or that specially crafted SVG or document uploads trigger unusual fetches or expose server file contents. In modern systems this often appears indirectly through libraries handling rich file types rather than through explicit XML APIs. Confirm that XML parsers, if present anywhere in the stack, disable dangerous entity resolution and do not automatically trust embedded external references. Also examine whether file upload paths route SVG or document formats through code that expands or interprets XML content.

# Cross-Site Scripting (XSS)

Inspect every place where user-controlled content can be rendered into HTML, SSR output, hydrated state, meta tags, email previews, admin panels, moderation tools, rich text displays, listing descriptions, organization names, profile fields, search pages, and any view that may transform or inject user text into the DOM. The core question is whether stored, reflected, or DOM-based XSS is possible when a user submits HTML, script-like payloads, event handlers, `javascript:` URLs, SVG content, markdown with embedded HTML, malformed attributes, or encoded variants that bypass simplistic filtering. In a Nuxt SSR application, the risk includes server-rendered output that already contains the malicious payload before it reaches the browser, as well as client-side rendering that uses dangerous HTML insertion mechanisms. A real bug may return HTTP 200 and save the payload successfully, then execute when another user, moderator, or crawler views the content, or it may reflect immediately in a search result or error message. Distinguish safe contextual encoding and robust sanitization from naive filtering that only strips obvious `<script>` tags while still allowing attribute-based execution, dangerous links, or execution inside special rendering contexts.

# HTTP Parameter Pollution

Review how query parameters, form inputs, repeated keys, and overlapping body-plus-query fields are parsed at every layer, including the browser, Cloudflare Worker, reverse proxy, FastAPI request parsing, and downstream business logic. This vulnerability appears when different components interpret duplicate parameters differently, allowing an attacker to pass validation with one value while the actual operation executes with another, or to alter access-control and routing logic through repeated or shadowed parameters. A practical example would be an endpoint where `organization_id` or `redirect` appears twice, and one layer reads the first occurrence while another consumes the last, enabling authorization bypass, open redirect, or mis-targeted actions. Another variant occurs when list-like inputs are accepted implicitly without strict schema rules, leading to unexpected filter expansion or hidden object selection. Confirm whether the framework rejects duplicate critical parameters, normalizes them safely, or at least ensures that every layer interprets the same canonical value for security-sensitive decisions.

# Host Header Injection

Assess whether the application uses the `Host` header or forwarded host information to construct absolute URLs, password reset links, invitation links, canonical URLs for SEO, redirect destinations, email content, tenant routing, or environment-specific behavior. A real vulnerability exists if an attacker can supply a malicious host that the application then reflects into trusted links or uses to influence security decisions, causing users to receive legitimate application emails containing attacker-controlled domains or causing SSR output to embed poisoned URLs. In stacks involving reverse proxies and Cloudflare, this risk often depends on whether the origin trusts forwarded host headers from the edge without an explicit allowlist. Review every place where absolute URLs are built, not just the obvious authentication flows, because preview links, unsubscribe links, invitation acceptance links, and SEO metadata frequently reuse the same helper functions. Confirm that only trusted hostnames are accepted and that host-derived logic cannot be manipulated from the public internet.

# Request Smuggling

Because traffic may traverse Cloudflare, edge workers, reverse proxies, and a FastAPI origin, review whether inconsistent HTTP parsing between layers could permit request smuggling, desynchronization, or cache confusion. The issue becomes exploitable when one component interprets request boundaries differently from another, often due to conflicting `Content-Length` and `Transfer-Encoding` handling, duplicate headers, unusual whitespace, or malformed but tolerated requests. A real exploit may allow an attacker to prepend or hide a second request, poison shared responses, bypass authentication on a following request, or cause one user’s request to be interpreted in the context of another. While some aspects depend on infrastructure behavior rather than application code alone, you should still inspect visible configuration, proxy header handling, framework assumptions, and worker request transformations. Document any evidence that multiple layers normalize requests differently, and distinguish between purely theoretical exposure and concrete signs that a malformed request could meaningfully alter backend interpretation or cache behavior.

# Server-Side Request Forgery (SSRF)

Inspect all features that cause the server or edge layer to fetch a URL, connect to a remote host, resolve metadata, build previews, import data, validate external resources, process callbacks, load remote images, or integrate with third-party services based on user input or semi-trusted configuration. SSRF is present when a caller can cause the backend or worker to make network requests to internal services, localhost, metadata endpoints, private IP ranges, or attacker-controlled targets in ways that expose data or alter internal state. A realistic bug might appear in preview generation, remote media import, webhook testing, map/geocoding integrations, or URL verification flows that accept arbitrary schemes or follow redirects too freely. Evidence can include unusual response content, differential error messages, timeout behavior, or code that passes a user-supplied URL directly into an HTTP client without strict allowlisting. Confirm whether internal address ranges, local hostnames, alternate IP notations, redirects, and DNS rebinding-style tricks are prevented, and whether the worker layer itself introduces SSRF potential by forwarding dynamic destinations.

# File Upload Vulnerabilities

Review all upload and attachment flows for images, avatars, listing assets, documents, imports, and any file-like object to determine whether file type validation, content validation, naming, storage, processing, and retrieval are secure. A real vulnerability may exist if the application trusts the file extension or MIME type alone, accepts active content such as HTML or SVG that is later served directly to browsers, allows double extensions or polyglot files, exposes private files through predictable paths, or processes attacker-controlled files through unsafe parsers or external tools. In practice, the issue may manifest as successful upload followed by execution in a browser context, unexpected server-side processing errors, public retrieval of a supposedly private file, or the ability to attach another user’s asset reference to one’s own object. Review how signed URLs are created and validated, whether file ownership and tenant ownership are enforced on download, and whether image processing, PDF preview, OCR, thumbnail generation, or metadata extraction expands the attack surface beyond mere storage.

# Path Traversal

Examine every place where file paths, filenames, archive entries, export names, attachment references, cache keys, or local storage paths are derived from user-controllable input. Path traversal becomes a real problem when an attacker can escape an intended directory boundary using sequences such as `../`, encoded separators, mixed slash variants, null-byte tricks, or crafted archive contents, thereby reading or overwriting unintended files. A realistic symptom is that a download or export endpoint behaves differently when given path-like values, or that unpacking an uploaded archive writes files outside the intended extraction directory. In environments with file previews, cached SSR artifacts, or local temp directories, traversal can also expose internal application files, configuration, or assets belonging to other tenants. Confirm whether all file paths are resolved against a strict allowlisted root using canonical path checks, whether archive extraction defends against Zip Slip style payloads, and whether any helper utilities concatenate path fragments unsafely.

# Open Redirect

Review every parameter and flow that influences post-action navigation, including `next`, `return_url`, `redirect`, `callback`, `continue`, login and logout flows, invitation acceptance, password reset, verification links, email links, and SSR route transitions. An open redirect exists when the application trusts a caller-supplied destination and redirects the user to an arbitrary external URL, which can support phishing, token theft in adjacent bugs, abuse of brand trust, and multi-step exploitation chains. A practical sign is an HTTP 302 or similar redirect to a foreign domain after submitting a crafted destination value, or the reflection of an attacker-controlled URL into a page or email link that appears legitimate because it originates from the application. Do not limit the review to explicit redirect endpoints; also inspect helper methods that build canonical navigation links or action completion flows. Confirm that redirects are restricted to relative paths or a strict allowlist of trusted domains, and that normalization does not allow scheme-relative, encoded, or obfuscated bypasses.

# Cache Poisoning

Analyze all caching layers, including Cloudflare, workers, SSR output caching, reverse proxies, browser cache directives, and any application-level cache, to determine whether untrusted request input can alter a cached response that is then served to other users. A severe issue exists if personalized or tenant-specific content can be cached under a shared key, or if attacker-controlled headers, query parameters, URL variants, or normalization quirks can poison a cache entry with content later delivered to victims. In a Nuxt SSR setup this is especially important because server-rendered pages may include data selected by authentication context, locale, role, or tenant membership. A real bug might cause user A’s private page state to appear to user B, or allow a malicious payload to be embedded into a shared cached HTML response. Review whether `Cache-Control` is correct for authenticated and semi-private content, whether cache keys incorporate every security-relevant dimension, whether workers strip or preserve sensitive headers appropriately, and whether any response with cookies or token-derived content can ever be stored in a shared cache.

# Insecure Deserialization

Inspect whether the application deserializes complex objects from untrusted or semi-trusted sources such as request bodies, cookies, signed tokens, queue messages, cached blobs, import files, or inter-service payloads without strict schema validation and safe handling. Insecure deserialization is present when attacker-controlled serialized data can create unexpected object graphs, invoke dangerous behavior, alter internal state, or bypass intended validation through type confusion or implicit reconstruction logic. In modern JSON-heavy applications this often appears less as classic object deserialization and more as unsafe trust in complex nested data structures that are rehydrated into internal models or execution contexts without defensive boundaries. A practical sign would be that unusual object shapes, extra fields, nested references, or encoded payloads cause side effects beyond ordinary validation errors. Determine whether the system only accepts strongly typed, allowlisted structures, whether signatures or integrity checks exist where needed, and whether any deserialized data influences control flow, permissions, filesystem access, or template rendering.

# Business Logic Flaws

Assess the business processes themselves, not only technical control points, because severe vulnerabilities often arise when each individual validation step appears correct but the overall workflow can be abused. Review invitation acceptance and cancellation, organization membership changes, ownership transfer, listing publication and moderation, edit-after-publish restrictions, account verification, password reset sequencing, soft-delete and restore, import and export, duplicate or clone operations, asset attachment, billing-related state, and any logic that depends on sequence, timing, or status transitions. A real flaw may allow a user to accept an invitation intended for another address, restore or edit an object after losing access, bypass moderation by racing two requests, or exploit inconsistent assumptions between one service and another. These bugs rarely reveal themselves through obvious error messages; instead they appear as successful state transitions the system should never allow. Pay particular attention to edge cases, race conditions, stale references, multi-step flows, and differences between the main UI path and alternate API calls.

# Resource Exhaustion (Logical DoS)

Even though missing rate limiting is explicitly out of scope, review whether the application can be abused through expensive but apparently legitimate operations that consume excessive CPU, memory, database time, external API budget, or storage. This includes extremely large page sizes, deeply nested filters, broad exports, repeated preview generation, oversized uploads, image and PDF transformations, OCR, analytics queries, expensive sorts, wide date ranges, and large batch operations. A real weakness exists if a single authenticated or unauthenticated user can trigger disproportionately costly work and receive normal success responses while degrading system availability or creating unacceptable infrastructure spend. This should be treated as logical resource exhaustion rather than generic missing throttling, and the report should tie the risk to specific endpoints, queries, or processing pipelines. Confirm whether request sizes, list sizes, export scopes, search complexity, processing concurrency, and expensive transformation paths are bounded by application logic, not merely by external edge controls.

# Information Disclosure

Inspect whether the application leaks information about users, organizations, records, environment details, or internal state through status codes, timing differences, side-channel behavior, redirects, metadata, enumeration-friendly identifiers, or inconsistent error messages. This is broader than direct sensitive data exposure because even small hints such as whether a record exists, whether an invitation is valid, whether a user email is registered, or whether a slug maps to a real object can materially help attackers enumerate the system and stage more serious abuse. A typical manifestation is one identifier returning 403 while a random identifier returns 404, revealing existence of protected objects, or an invitation endpoint confirming whether a token is valid before requiring authentication. In slug-based systems, disclosure can also occur if the suffix is predictable or if invalid slug text still resolves to a real object based solely on the trailing token. Document how much an attacker can learn without authorization and whether that information meaningfully lowers the barrier to exploitation.

# Improper Error Handling

Review how the application handles exceptions, validation failures, authorization denials, database errors, parser errors, dependency failures, and worker-origin mismatches, and determine whether those failures leak technical details or create inconsistent security behavior. A serious issue exists if the API or SSR output exposes stack traces, SQL errors, file paths, hostnames, library internals, token fragments, serializer dumps, or raw exception messages to clients. Another common problem is inconsistency, where one endpoint properly returns a generic denial while another reveals why access failed, thereby enabling enumeration or targeted exploitation. In FastAPI-based applications, unhandled exceptions and verbose validation output can be especially revealing if not normalized by a secure error handler. Review not only the visible response bodies but also headers, debug pages, content type changes, and worker error wrappers, because those often disclose environment information or underlying implementation details useful to an attacker.

# CORS Misconfiguration

Assess the full CORS policy at the API, edge, and any SSR-related data endpoints to determine whether cross-origin access is broader than necessary, whether credentials are allowed from overly wide origins, and whether origin reflection or wildcard behavior weakens the security model. A real problem exists if the backend dynamically reflects arbitrary origins into `Access-Control-Allow-Origin`, especially in combination with credentialed requests, or if sensitive endpoints are unnecessarily readable from external web applications. Even in bearer-token systems, permissive CORS can enable abuse when tokens are accessible in browser contexts, when hybrid cookie flows exist, or when internal APIs are assumed to be callable only from the first-party frontend. Review preflight handling, allowed headers, exposed headers, method lists, and whether worker logic differs from origin logic. Confirm that only expected origins are trusted, that credentials are used narrowly if at all, and that the effective browser policy matches the intended architecture rather than a permissive development convenience.

# Clickjacking

Determine whether important application views can be embedded in an attacker-controlled frame and whether the application provides sufficient framing restrictions for sensitive pages such as login, settings, publishing flows, membership changes, administrative views, billing steps, and confirmation screens. Clickjacking becomes meaningful when a victim can be tricked into interacting with a hidden or disguised instance of the real application while authenticated, resulting in unintended state changes or approvals. The observable symptom is usually not an API anomaly but the absence of defensive headers or policies that prevent framing, combined with workflows that can be triggered through ordinary clicks. In SSR-heavy applications, attention should also be paid to rendered pages that perform important actions after minimal user interaction, because those are especially susceptible to UI deception. Confirm whether frame restrictions are enforced consistently across all sensitive routes and whether any edge or worker layer strips or overrides those protections.

# Missing Security Headers

Review whether the application consistently emits appropriate security headers on HTML, SSR responses, static assets where relevant, and API responses where meaningful. This includes checking for a properly designed Content Security Policy, framing protections, MIME sniffing prevention, referrer restrictions, transport security, and cache directives that prevent sensitive pages from being stored in the wrong place. Missing headers do not always constitute an exploit by themselves, but they materially increase the impact and reliability of other vulnerabilities such as XSS, clickjacking, mixed-content problems, or information leakage through referrers and caches. A practical concern is when HTML pages carrying authenticated content are delivered without CSP and framing restrictions, or when private content lacks cache controls and is eligible for storage in intermediate layers. Confirm not just presence but correctness, because misconfigured CSP or contradictory header behavior across origin and edge can create a false sense of security.

# Insufficient Logging & Monitoring

Review whether the application produces enough security-relevant audit data to investigate abuse, privilege changes, invitation use, ownership transfer, authentication events, suspicious access denials, file access, publication actions, and destructive operations, while also ensuring that logs do not themselves become a source of sensitive data leakage. A meaningful weakness exists if critical actions leave no audit trail, making incident response impossible, or if tokens, personal data, raw request bodies, signed URLs, internal secrets, or private content are logged in plaintext. In distributed environments involving workers, edge caches, SSR, and a FastAPI backend, security visibility often fails because each layer logs partial context without a coherent identifier trail. Evaluate whether high-risk operations are attributable to a user and tenant, whether authentication anomalies are visible, whether access-control failures are distinguishable from ordinary errors, and whether log content is appropriately minimized and protected. The goal is not generic observability commentary but a concrete assessment of whether abuse would be detectable without creating a secondary confidentiality problem.
```
